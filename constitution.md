# Todo App Constitution
## Evolution of Todo - Hackathon II

---

## Project Mission
Build a todo application that evolves from a simple console app to a cloud-native AI chatbot through 5 progressive phases, using **Spec-Driven Development** with Claude Code and Spec-Kit Plus.

---

## Core Principles

### 1. Spec-Driven Development (MANDATORY)
- **NO MANUAL CODING ALLOWED**: All code must be generated by Claude Code from specifications.
- Every feature must have a written specification in `/specs` before implementation.
- Specifications must follow the standard template in `.spec-kit/templates/spec-template.md`.
- Code generation is iterative: refine specs until Claude Code produces correct output.

### 2. Architecture of Intelligence
- **Reusable First**: Build reusable skills, templates, and sub-agents before writing feature code.
- **Progressive Evolution**: Each phase builds on the previous, maintaining backward compatibility.
- **Modularity**: Design for future phases - Phase 1 in-memory storage must easily migrate to Phase 2 database.
- **Separation of Concerns**: Clear boundaries between data models, business logic, and presentation.

### 3. Quality Standards
- **Testing**: Every feature must have corresponding tests (Pytest).
- **Documentation**: README.md and CLAUDE.md must be kept current.
- **Code Review**: Generated code must be validated against specs before acceptance.
- **Clean Code**: Follow PEP 8, type hints, and docstrings for all functions.

---

## Technology Stack

### Phase 1: Python Console App
- **Language**: Python 3.13+
- **Package Manager**: UV (for fast dependency management)
- **Data Validation**: Pydantic v2 (for type-safe models)
- **Testing**: Pytest
- **Storage**: In-memory (Python dict/list)
- **CLI**: Rich (for beautiful terminal output)

### Phase 2+: Full-Stack Evolution (Future)
- **Backend**: FastAPI, SQLModel
- **Database**: Neon Serverless PostgreSQL
- **Frontend**: Next.js 16+, Better Auth
- **AI**: OpenAI Agents SDK, Official MCP SDK
- **Cloud**: Kubernetes (Minikube → Oracle Cloud OKE)
- **Events**: Kafka (Redpanda Cloud / Strimzi)
- **Runtime**: Dapr

## Data Model (Phase 1)

### Task
```python
class Task(BaseModel):
    id: int                                    # Auto-generated
    title: str                                 # Required, 1-200 chars
    description: str | None                    # Optional, max 1000 chars
    status: TaskStatus                         # PENDING | COMPLETED
    created_at: datetime                       # Auto-generated
    updated_at: datetime                       # Auto-updated
```

## Phase 1 Features (Basic Level)

### Must Implement
1. **Add Task**: Create new todo items with title and optional description
2. **Delete Task**: Remove tasks by ID
3. **Update Task**: Modify task title and/or description
4. **View Task List**: Display all tasks with status indicators
5. **Mark as Complete**: Toggle task completion status

## Development Workflow

1. Write Specification using template
2. Validate Specification with Architect skill
3. Generate Tests from spec
4. Generate Code via Claude Code
5. Validate & Iterate until tests pass
6. Backup Phase when complete

---

## Backup Strategy (IMPORTANT!)

When a phase is complete:
```bash
# Copy entire working code to Phases/Phase_1/
# From Windows: Copy-Item D:\Todo_App\* D:\Todo_App\Phases\Phase_1\ -Recurse
# Push to GitHub
git add . && git commit -m "feat(phase1): Complete" && git push
```

## Rules & Constraints

### MUST DO ✅
- Write specs before code
- Use Claude Code for all code generation
- Follow Pydantic models for data validation
- Write tests for every feature
- Use UV for dependency management
- Keep constitution.md updated

### MUST NOT DO ❌
- Write code manually (except spec files)
- Skip specification step
- Deploy without tests passing
- Hardcode values
- Mix business logic with presentation layer
